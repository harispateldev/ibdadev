# Ibda Dev design rules

Ibda Dev is not a website-only studio. Treat it as a modern business technology agency that designs and builds mobile apps, web platforms, SaaS products, AI systems, dashboards, commerce experiences, automations, and complete digital solutions.

Brand system:

- Primary brand name: Ibda Dev.
- Primary tagline: Software. AI. Automation.
- Core line: Ibda Dev designs and builds modern software, AI products, and automation systems.
- Footer line: Ibda Dev — Software. AI. Automation.
- Brand tone: minimal, premium, symmetric, structured, and modern.
- Logo usage: use the black-and-gold geometric Ibda Dev mark from `public/brand/ibda-dev-mark.png` in header, footer, and brand moments where the mark has enough space to read clearly.

Follow these rules for every new improvement:

- Start with the business purpose before the visual. Each section must answer what Ibda Dev does, who it helps, what problem it solves, or what outcome the client gets.
- Design for both clients and developers. The site should feel premium, clear, trustworthy, and useful to business buyers while still showing strong technical craft.
- Keep the site modern, cool, smooth, and eye-catching, but never add motion or visuals only because they look trendy.
- Use current market references as inspiration for structure, interaction, and motion only. Do not mention inspiration sites in UI copy.
- Make mobile apps a first-class offer alongside web, AI, SaaS, dashboards, commerce, and automation.
- Replace abstract decoration with visual proof: product surfaces, mobile screens, workflow states, system maps, outcome panels, credible project previews, or client/business results.
- Keep copy short and sharp. Prefer fast-scanning headlines, one-line statements, concise labels, and clear calls to action over paragraphs and repeated chips.
- Give every section a distinct concept. Do not repeat the same browser-window, dashboard card, or product-panel composition across the whole page.
- Use smooth motion to explain state, progress, flow, hierarchy, or interaction: device float, live pulse, scan line, progress fill, reveal, hover depth, subtle parallax, or capability ribbons.
- Motion must feel premium, not noisy. Keep animations calm, performant, responsive, and covered by reduced-motion support.
- Scroll-section animation should be controlled by scroll progress where possible, not fast one-shot `whileInView` reveals that complete before the user feels the movement.
- Keep looped loader/skeleton-style effects rare. Use them only when they communicate live system state, progress, or product activity.
- Backgrounds can be rich, but they must not overpower the main message, section titles, service proof, or calls to action.
- Keep the Work page reserved for real projects. Until real projects are ready, use clean empty project slots and do not invent fake case studies, project names, screenshots, or outcomes.
- About and Work must match the main site quality, brand tone, and business-facing clarity.
- Before finishing a visual change, verify desktop and mobile views for text fit, overlap, hierarchy, scroll feel, and whether the section story is obvious in one sentence.

Decision test:

If a visual, animation, or section cannot be explained in one clear business-facing sentence, redesign it before shipping.
